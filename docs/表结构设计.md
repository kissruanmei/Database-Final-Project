# 逻辑结构设计 — 表结构

## 一、E-R 图转关系模式

将 E-R 图中的实体和联系转换为关系模式（表），规则如下：
- 每个实体 → 一张表
- 1:N 联系 → 在 N 端表中加入 1 端的主键作为外键
- 1:1 联系 → 在任一方加入另一方主键作为外键

### 转换结果

| 关系模式 | 来源 | 说明 |
|----------|------|------|
| Owner(车主) | 实体 | 独立成表 |
| Vehicle(车辆) | 实体 + "拥有"联系 | 加入 owner_id 外键 |
| Permit(通行证) | 实体 + "持有"联系 + "审批"联系 | 加入 plate_number 外键 + admin_id 外键 |
| Violation(违规记录) | 实体 + "产生"联系 + "录入"联系 | 加入 plate_number 外键 + admin_id 外键 |
| Penalty(处罚记录) | 实体 + "对应"联系 + "处理"联系 | 加入 violation_id 外键 + admin_id 外键 |
| Admin(管理员) | 实体 | 独立成表 |

## 二、表结构定义

### 表 1：车主信息表（Owner）

| 列名 | 数据类型 | 约束 | 说明 |
|------|----------|------|------|
| owner_id | INT | PRIMARY KEY, IDENTITY(1,1) | 车主编号，自增主键 |
| owner_name | VARCHAR(50) | NOT NULL | 车主姓名 |
| owner_type | VARCHAR(20) | NOT NULL, CHECK(IN('教职工','学生','临时访客')) | 车主类型 |
| id_card | VARCHAR(18) | NOT NULL, UNIQUE | 身份证号 |
| phone | VARCHAR(11) | NOT NULL | 联系电话 |
| department | VARCHAR(100) | | 所属部门/学院，临时访客可为空 |
| total_points | INT | NOT NULL, DEFAULT 0, CHECK(>=0) | 累计违规扣分 |

### 表 2：车辆信息表（Vehicle）

| 列名 | 数据类型 | 约束 | 说明 |
|------|----------|------|------|
| plate_number | VARCHAR(10) | PRIMARY KEY | 车牌号 |
| vehicle_type | VARCHAR(20) | NOT NULL, CHECK(IN('小型汽车','大型汽车','摩托车')) | 车辆类型 |
| vehicle_color | VARCHAR(10) | NOT NULL | 车辆颜色 |
| owner_id | INT | NOT NULL, FOREIGN KEY → Owner(owner_id) | 所属车主编号 |

### 表 3：通行证表（Permit）

| 列名 | 数据类型 | 约束 | 说明 |
|------|----------|------|------|
| permit_id | INT | PRIMARY KEY, IDENTITY(1,1) | 通行证编号，自增主键 |
| plate_number | VARCHAR(10) | NOT NULL, FOREIGN KEY → Vehicle(plate_number) | 对应车牌号 |
| permit_type | VARCHAR(20) | NOT NULL, CHECK(IN('全校通行','限时通行','指定区域')) | 权限类型 |
| start_date | DATE | NOT NULL | 生效日期 |
| end_date | DATE | NOT NULL, CHECK(> start_date) | 到期日期 |
| permit_status | VARCHAR(10) | NOT NULL, DEFAULT '有效', CHECK(IN('有效','过期','注销')) | 通行证状态 |
| admin_id | INT | NOT NULL, FOREIGN KEY → Admin(admin_id) | 审批管理员 |

### 表 4：违规记录表（Violation）

| 列名 | 数据类型 | 约束 | 说明 |
|------|----------|------|------|
| violation_id | INT | PRIMARY KEY, IDENTITY(1,1) | 违规编号，自增主键 |
| plate_number | VARCHAR(10) | NOT NULL, FOREIGN KEY → Vehicle(plate_number) | 违规车牌号 |
| violation_type | VARCHAR(50) | NOT NULL | 违规类型 |
| violation_time | DATETIME | NOT NULL | 违规时间 |
| violation_location | VARCHAR(100) | NOT NULL | 违规地点 |
| violation_desc | TEXT | | 违规描述，可为空 |
| points_deducted | INT | NOT NULL, CHECK(>0) | 扣减分数 |
| admin_id | INT | NOT NULL, FOREIGN KEY → Admin(admin_id) | 录入管理员 |

### 表 5：处罚记录表（Penalty）

| 列名 | 数据类型 | 约束 | 说明 |
|------|----------|------|------|
| penalty_id | INT | PRIMARY KEY, IDENTITY(1,1) | 处罚编号，自增主键 |
| violation_id | INT | NOT NULL, UNIQUE, FOREIGN KEY → Violation(violation_id) | 对应违规编号 |
| penalty_type | VARCHAR(50) | NOT NULL, CHECK(IN('警告','罚款','暂停通行证','禁止入校')) | 处罚类型 |
| penalty_amount | DECIMAL(10,2) | DEFAULT 0, CHECK(>=0) | 罚款金额 |
| penalty_status | VARCHAR(10) | NOT NULL, DEFAULT '待处理', CHECK(IN('待处理','已处理','已申诉','申诉通过')) | 处罚状态 |
| process_time | DATETIME | | 处理时间，未处理时为空 |
| admin_id | INT | FOREIGN KEY → Admin(admin_id) | 处理管理员 |

### 表 6：管理员表（Admin）

| 列名 | 数据类型 | 约束 | 说明 |
|------|----------|------|------|
| admin_id | INT | PRIMARY KEY, IDENTITY(1,1) | 管理员编号，自增主键 |
| admin_name | VARCHAR(50) | NOT NULL, UNIQUE | 管理员姓名 |
| admin_password | VARCHAR(64) | NOT NULL | 登录密码 |

## 三、外键关系汇总

| 外键所在表 | 外键列 | 引用表 | 引用列 | 行为 |
|-----------|--------|--------|--------|------|
| Vehicle | owner_id | Owner | owner_id | ON DELETE CASCADE |
| Permit | plate_number | Vehicle | plate_number | ON DELETE CASCADE |
| Permit | admin_id | Admin | admin_id | ON DELETE NO ACTION |
| Violation | plate_number | Vehicle | plate_number | ON DELETE CASCADE |
| Violation | admin_id | Admin | admin_id | ON DELETE NO ACTION |
| Penalty | violation_id | Violation | violation_id | ON DELETE CASCADE |
| Penalty | admin_id | Admin | admin_id | ON DELETE NO ACTION |

> 说明：Vehicle/Violation/Penalty 的 owner_id/plate_number/violation_id 使用 CASCADE（级联删除），admin_id 使用 NO ACTION（管理员删除不影响历史记录）。

## 四、规范化分析

### 第一范式（1NF）
- 所有表的每个字段都是**原子的**，不可再分
- 例如：phone 只存一个号码，不存多个；department 只存一个部门
- **满足 1NF**

### 第二范式（2NF）
- 所有表的主键均为**单列主键**（owner_id / plate_number / permit_id / violation_id / penalty_id / admin_id）
- 不存在组合主键，因此不存在"部分依赖"问题
- **满足 2NF**

### 第三范式（3NF）
逐表检查非主属性对主键的传递依赖：

| 表 | 检查 | 结论 |
|----|------|------|
| Owner | owner_name, phone, department 等均直接依赖 owner_id，不存在 owner_type → department 的依赖（同类型车主可以属于不同部门） | 无传递依赖 |
| Vehicle | vehicle_type, vehicle_color 直接依赖 plate_number | 无传递依赖 |
| Permit | permit_type, start_date, end_date, status 均直接依赖 permit_id | 无传递依赖 |
| Violation | 所有属性直接依赖 violation_id | 无传递依赖 |
| Penalty | 所有属性直接依赖 penalty_id | 无传递依赖 |
| Admin | admin_name, admin_password 直接依赖 admin_id | 无传递依赖 |

**结论：所有表均满足 3NF。**

## 五、关系模式总览（简洁版）

```
Owner(owner_id, owner_name, owner_type, id_card, phone, department, total_points)
  PK: owner_id  UK: id_card

Vehicle(plate_number, vehicle_type, vehicle_color, owner_id)
  PK: plate_number  FK: owner_id → Owner

Permit(permit_id, plate_number, permit_type, start_date, end_date, permit_status, admin_id)
  PK: permit_id  FK: plate_number → Vehicle, admin_id → Admin

Violation(violation_id, plate_number, violation_type, violation_time, violation_location, violation_desc, points_deducted, admin_id)
  PK: violation_id  FK: plate_number → Vehicle, admin_id → Admin

Penalty(penalty_id, violation_id, penalty_type, penalty_amount, penalty_status, process_time, admin_id)
  PK: penalty_id  UK: violation_id  FK: violation_id → Violation, admin_id → Admin

Admin(admin_id, admin_name, admin_password)
  PK: admin_id  UK: admin_name
```
