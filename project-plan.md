## 一、数据库应用系统设计实现步骤

### 总体流程（6 步）

| 步骤 | 基本任务 | 涉及知识/方法/工具 |
|------|----------|-------------------|
| 1. 需求分析 | 明确系统要管什么：车辆注册、违规记录、处罚处理、数据查询；画出数据流图(DFD)、编写数据字典 | 需求调研方法、DFD图、数据字典编写 |
| 2. 概念结构设计 | 设计 E-R 图：识别实体（车主、车辆、违规记录等）、属性、实体间关系 | E-R 模型、实体/属性/联系的识别 |
| 3. 逻辑结构设计 | 将 E-R 图转换为关系模式（表结构），确定主键/外键、规范化到 3NF | 关系模型、E-R→关系转换规则、范式理论(1NF/2NF/3NF) |
| 4. 物理结构设计 | 在 SQL Server 中建库建表，设计索引、约束，考虑性能 | SQL Server Management Studio (SSMS)、T-SQL DDL 语句、索引策略 |
| 5. 数据库实施与应用开发 | 导入测试数据、编写增删改查操作、实现业务逻辑（存储过程、视图、触发器）、开发 CLI 界面 | T-SQL（DML/存储过程/视图/触发器）、Python + pyodbc、CLI 界面开发 |
| 6. 系统测试与验收 | 功能测试、数据完整性测试、准备答辩演示 | 测试用例设计、答辩PPT/文档准备 |

### 各步骤详细说明

**步骤 1 — 需求分析**
- 与"虚拟客户"（你自己模拟保卫处需求）梳理业务流程
- 输出：需求说明文档 + 数据流图(DFD) + 数据字典
- 工具：纸笔 / draw.io / ProcessOn 画图

**步骤 2 — 概念结构设计**
- 识别核心实体：车主、车辆、通行证(权限)、违规记录、处罚记录、管理员
- 画 E-R 图（实体-属性-联系图）
- 工具：draw.io、PowerDesigner 或 Visio

**步骤 3 — 逻辑结构设计**
- E-R 图 → 关系模式（表清单）
- 检查规范化（消除冗余，达到 3NF）
- 输出：各表的字段、类型、主键、外键定义

**步骤 4 — 物理结构设计**
- 在 SSMS 中创建数据库和表
- 添加约束（NOT NULL、UNIQUE、CHECK、DEFAULT、FOREIGN KEY）
- 为高频查询字段建索引

**步骤 5 — 实施与开发**
- 插入测试数据（每表 10-20 条）
- 编写视图、存储过程、触发器
- 用 Python (pyodbc) 开发 CLI 交互界面

**步骤 6 — 测试与验收**
- 功能走查、边界测试
- 准备课程设计报告 + 答辩演示

## 二、已有基础 & 问题清单

### 已学习/掌握的内容

### 问题清单
1. ~~存储过程、视图、触发器 是否已经学过？~~ → 确认未学过，需要自学
2. Python 连接 SQL Server（pyodbc）从未用过，需要学习
3. DFD 数据流图没有画过，需要学习基本画法
4. E-R 图的规范画法不够熟练
5. 测试数据如何构造有代表性的数据集

### 主要难点
- **难点 1（最大难点）**：存储过程/触发器未学过，需要从零自学并在项目中实际应用
- **难点 2**：数据库规范化 — 确保设计满足 3NF 同时不丢失业务语义
- **难点 3**：多表关联查询的性能和正确性
- **难点 4**：Python CLI 与 SQL Server 的稳定连接和异常处理

### 自学安排（针对未学内容）

因为存储过程、视图、触发器都没学过，但又是答辩加分项，建议在第2周穿插自学：

| 知识点 | 自学时间 | 学习内容 | 建议资源 |
|--------|----------|----------|----------|
| **视图** | 第2周第8天，约1h | CREATE VIEW 语法、多表视图、在查询中使用视图 | B站搜"SQL Server 视图教程" |
| **存储过程** | 第2周第9-10天，约3h | CREATE PROCEDURE、输入/输出参数、IF/WHILE逻辑、EXEC 调用 | B站搜"SQL Server 存储过程" |
| **触发器** | 第2周第11天，约2h | CREATE TRIGGER、AFTER/INSTEAD OF、inserted/deleted 虚拟表 | B站搜"SQL Server 触发器" |

**学习策略**：先做基础功能（建表+CRUD），再逐步加入视图→存储过程→触发器。不需要三个都精通，能写出项目用到的那几个即可。

## 三、拓展部分需求（初步）

> 基础功能必须做，拓展部分是加分项，按精力酌情选取。

| 拓展方向 | 具体内容 | 难度 |
|----------|----------|------|
| 视图(View) | 常用多表查询封装为视图（如"某车主违规历史"、"本月违规统计"） | ★☆☆（自学1h） |
| 存储过程 | 处理违规流程（自动记录、累计积分、触发处罚）；注册审批流程 | ★★☆（自学3h） |
| 触发器 | 违规记录插入时自动更新车主违规积分；通行证到期自动失效 | ★★☆（自学2h） |
| 统计报表 | 按月/学院/违规类型统计违规数量，输出报表 | ★☆☆ |
| 数据完整性约束 | CHECK 约束（车牌号格式、手机号格式）、级联操作 | ★☆☆ |
| 权限管理 | 不同角色（管理员/车主）有不同操作权限 | ★★☆ |
| 数据备份与恢复 | 演示数据库备份还原操作 | ★☆☆ |

**建议选取**：视图 + 存储过程 + 触发器（这三个最能体现数据库课程知识点，答辩加分明显）

### 协作点（需要两人共同完成）
- 需求分析阶段一起讨论业务流程
- E-R 图一起审核确认    
- 联调测试阶段一起验证
- 答辩演示一起准备

### 分工说明
- A 偏"数据库侧"：所有 SQL 相关的工作
- B 偏"应用侧"：Python 代码 + 文档报告
- 这样分工的好处是：各自专注一个领域，减少代码冲突，联调时只需对接数据接口
